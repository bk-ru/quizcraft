Ниже - декомпозиция под **минимальную рабочую версию с LM Studio** как единственным провайдером модели.
Технологические допущения для этой декомпозиции: backend на Python с типизированными контрактами API, генерация через **LM Studio OpenAI-compatible API** на локальном адресе `http://localhost:1234/v1`, структурированный ответ через `/v1/chat/completions` с JSON Schema, RAG во второй очереди через `/v1/embeddings`, экспорт через `python-docx` и `python-pptx`, тесты на `pytest`, логирование через стандартный `logging`. Это соответствует официальной документации LM Studio, FastAPI/Pydantic, `python-docx`, `python-pptx`, `pytest` и Python `logging`. ([LM Studio][1])

---

## 0. Границы MVP

MVP = **локальный однопользовательский сервис** со следующими возможностями:

* загрузка одного документа;
* парсинг PDF/DOCX/TXT;
* прямая генерация квиза через LM Studio;
* структурированный результат в JSON;
* базовое редактирование результата;
* экспорт в JSON;
* обработка ошибок, логирование, pytest-тесты.

RAG, DOCX/PPTX, регенерация отдельных вопросов, несколько сценариев генерации, профили моделей - во второй очереди.

# 1. Задачи по модулям

## 1. API / backend

### BE-001 [MVP] Инициализация структуры backend-проекта

* **Описание:** создать каркас backend-приложения с модулями `api`, `core`, `domain`, `services`, `clients`, `repositories`, `tests`.
* **Вход:** нет.
* **Выход:** пустой, но запускаемый backend-проект.
* **DoD:** проект стартует локально; все модули подключены; есть единая точка входа приложения.

### BE-002 [MVP] Модель конфигурации приложения

* **Описание:** реализовать централизованную загрузку настроек из env-файла и переменных окружения.
* **Вход:** `LM_STUDIO_BASE_URL`, `LM_STUDIO_MODEL`, `REQUEST_TIMEOUT`, `MAX_FILE_SIZE_MB`, `LOG_LEVEL`.
* **Выход:** объект конфигурации, доступный во всех слоях приложения.
* **DoD:** приложение стартует без хардкода настроек; при отсутствии обязательных переменных поднимается контролируемая ошибка конфигурации.

### BE-003 [MVP] Единая схема ошибок backend

* **Описание:** ввести иерархию доменных исключений и единый формат error-response API.
* **Вход:** исключения из parser/generator/export/client слоев.
* **Выход:** JSON-ответ ошибки с кодом, сообщением, техническим идентификатором.
* **DoD:** для всех ожидаемых ошибок backend возвращает единообразный ответ; traceback не утекает пользователю.

### BE-004 [MVP] Endpoint проверки доступности backend

* **Описание:** реализовать endpoint healthcheck приложения.
* **Вход:** HTTP GET.
* **Выход:** статус backend и базовая информация о конфигурации режима.
* **DoD:** endpoint возвращает 200 при живом приложении и не зависит от LM Studio.

### BE-005 [MVP] Endpoint проверки доступности LM Studio

* **Описание:** реализовать endpoint проверки связи backend -> LM Studio.
* **Вход:** HTTP GET/POST без пользовательских данных.
* **Выход:** статус доступности LM Studio, модель по умолчанию, диагностическое сообщение.
* **DoD:** при недоступности LM Studio возвращается контролируемая ошибка 503 с понятным текстом.

### BE-006 [MVP] Endpoint загрузки документа

* **Описание:** реализовать endpoint приема одного файла с валидацией размера и типа.
* **Вход:** файл `pdf|docx|txt`.
* **Выход:** `document_id`, метаданные файла.
* **DoD:** неподдерживаемые расширения и пустые файлы отклоняются; корректный файл сохраняется и получает `document_id`.

### BE-007 [MVP] Endpoint генерации квиза

* **Описание:** реализовать endpoint генерации квиза по `document_id` в прямом режиме.
* **Вход:** `document_id`, число вопросов, язык, сложность, тип квиза.
* **Выход:** `quiz_id`, структура квиза.
* **DoD:** endpoint вызывает полный pipeline direct-generation; ответ валиден по схеме; ошибки LM Studio и ошибки схемы обрабатываются корректно.

### BE-008 [MVP] Endpoint чтения квиза

* **Описание:** реализовать получение ранее сгенерированного квиза по `quiz_id`.
* **Вход:** `quiz_id`.
* **Выход:** структура квиза.
* **DoD:** квиз возвращается в том виде, в котором хранится после валидации и нормализации.

### BE-009 [MVP] Endpoint сохранения изменений в квизе

* **Описание:** реализовать обновление квиза после ручного редактирования.
* **Вход:** `quiz_id`, измененная структура квиза.
* **Выход:** сохраненный квиз.
* **DoD:** обновление проходит только после валидации структуры; некорректные данные отклоняются.

### BE-010 [MVP] Endpoint экспорта в JSON

* **Описание:** реализовать экспорт квиза в JSON-файл.
* **Вход:** `quiz_id`.
* **Выход:** файл `.json`.
* **DoD:** файл скачивается, соответствует внутренней схеме, имя файла формируется детерминированно.

### BE-011 [P2] Endpoint экспорта в DOCX

* **Описание:** реализовать экспорт квиза в DOCX.
* **Вход:** `quiz_id`, шаблон экспорта.
* **Выход:** файл `.docx`.
* **DoD:** документ открывается в Word/LibreOffice; структура вопросов и вариантов сохранена.

### BE-012 [P2] Endpoint экспорта в PPTX

* **Описание:** реализовать экспорт квиза в PPTX.
* **Вход:** `quiz_id`, шаблон презентации.
* **Выход:** файл `.pptx`.
* **DoD:** презентация открывается; на каждом слайде корректно размещен вопрос и варианты ответа.

### BE-013 [P2] Endpoint регенерации одного вопроса

* **Описание:** реализовать пересоздание только одного вопроса внутри квиза.
* **Вход:** `quiz_id`, `question_id`, режим регенерации.
* **Выход:** новая версия вопроса.
* **DoD:** обновляется только один вопрос; остальные данные квиза не меняются.

## 2. Работа с моделями / LM Studio

### LM-001 [MVP] Абстракция провайдера модели

* **Описание:** ввести интерфейс `LLMProvider` с методами `healthcheck`, `generate_structured`, `embed`.
* **Вход:** доменные модели запроса.
* **Выход:** унифицированный результат генерации/эмбеддингов.
* **DoD:** backend не знает о деталях LM Studio; все вызовы идут через интерфейс провайдера.

### LM-002 [MVP] Клиент LM Studio для chat completions

* **Описание:** реализовать HTTP-клиент LM Studio через OpenAI-compatible endpoint `/v1/chat/completions`.
* **Вход:** system prompt, user prompt, schema, model, inference params.
* **Выход:** структурированный ответ модели.
* **DoD:** клиент умеет передавать JSON Schema и получать валидный JSON-ответ. ([LM Studio][1])

### LM-003 [MVP] Healthcheck клиента LM Studio

* **Описание:** реализовать проверку доступности локального сервера LM Studio.
* **Вход:** base URL.
* **Выход:** статус `available|unavailable` и текст диагностики.
* **DoD:** ошибки соединения, timeout и invalid response различаются и маппятся в свои исключения.

### LM-004 [MVP] Обертка над retry/timeout для вызовов LM Studio

* **Описание:** реализовать единый механизм таймаутов, лимита повторов и обратнойoff-паузы.
* **Вход:** вызываемая функция, max retries, timeout.
* **Выход:** результат вызова или доменная ошибка.
* **DoD:** временные ошибки повторяются; постоянные ошибки не ретраятся; retry покрыт тестами.

### LM-005 [P2] Клиент LM Studio для embeddings

* **Описание:** реализовать клиент для `/v1/embeddings`.
* **Вход:** список строк.
* **Выход:** список embedding-векторов.
* **DoD:** endpoint возвращает эмбеддинги; ошибки модели эмбеддингов и ошибки схемы различаются. ([LM Studio][2])

### LM-006 [P2] Поддержка выбора модели LM Studio на запрос

* **Описание:** реализовать выбор модели из конфигурации запроса, а не только глобальной default-модели.
* **Вход:** `model_name`.
* **Выход:** генерация на выбранной модели.
* **DoD:** backend валидирует whitelist доступных моделей и не принимает произвольное имя без разрешения.

### LM-007 [Future] Адаптер Ollama

* **Описание:** добавить реализацию `LLMProvider` для Ollama.
* **Вход:** те же доменные запросы, что для LM Studio.
* **Выход:** совместимый ответ.
* **DoD:** переключение провайдера не требует изменений в orchestration-слое.

### LM-008 [Future] Адаптер внешнего API

* **Описание:** добавить реализацию `LLMProvider` для OpenRouter/платного API.
* **Вход:** те же доменные запросы.
* **Выход:** совместимый ответ.
* **DoD:** backend поддерживает выбор провайдера через конфиг и feature flag.

## 3. Доменная модель и схема результата

### DM-001 [MVP] Доменные модели квиза

* **Описание:** описать модели `Quiz`, `Question`, `Option`, `Explanation`, `GenerationRequest`, `GenerationResult`.
* **Вход:** нет.
* **Выход:** единый набор доменных классов/схем.
* **DoD:** все внутренние модули работают через эти модели, а не через сырые dict.

### DM-002 [MVP] JSON Schema результата генерации

* **Описание:** зафиксировать строгую схему ответа модели.
* **Вход:** требования к квизу.
* **Выход:** JSON Schema.
* **DoD:** схема покрывает обязательные поля, ограничения по длине, допустимые типы вопросов.

### DM-003 [MVP] Нормализатор результата модели

* **Описание:** привести выход модели к каноническому виду.
* **Вход:** сырой JSON модели.
* **Выход:** нормализованная структура квиза.
* **DoD:** лишние поля отбрасываются, строки триммятся, пустые варианты удаляются, индексы ответов приводятся к единому формату.

### DM-004 [MVP] Валидатор бизнес-правил квиза

* **Описание:** проверить не только JSON Schema, но и доменные правила.
* **Вход:** структура квиза.
* **Выход:** список нарушений или валидный объект.
* **DoD:** валидируются: число вопросов, наличие хотя бы одного правильного ответа, отсутствие дублей вариантов, непустой текст вопроса.

## 4. Парсинг документов

### PR-001 [MVP] Валидатор входного файла

* **Описание:** реализовать проверку расширения, MIME-типа, размера, пустого содержимого.
* **Вход:** uploaded file.
* **Выход:** validated file descriptor.
* **DoD:** отклоняются неподдерживаемые расширения, нулевой размер, превышение лимита.

### PR-002 [MVP] Парсер TXT

* **Описание:** извлечение текста из `.txt`.
* **Вход:** txt file.
* **Выход:** raw text.
* **DoD:** поддерживаются UTF-8 и fallback-декодирование; при битом файле поднимается контролируемая ошибка.

### PR-003 [MVP] Парсер DOCX

* **Описание:** извлечение текста из `.docx`.
* **Вход:** docx file.
* **Выход:** raw text.
* **DoD:** извлекаются абзацы в читаемом порядке; пустой документ распознается как ошибка.

### PR-004 [MVP] Парсер PDF

* **Описание:** извлечение текста из `.pdf`.
* **Вход:** pdf file.
* **Выход:** raw text.
* **DoD:** текст извлекается постранично; PDF без извлекаемого текста возвращает понятную ошибку.

### PR-005 [MVP] Нормализация текста документа

* **Описание:** очистить текст от лишних переносов, повторных пробелов, мусорных символов.
* **Вход:** raw text.
* **Выход:** normalized text.
* **DoD:** нормализованный текст стабилен для одинаковых входных данных; правило нормализации покрыто тестами.

### PR-006 [MVP] Извлечение метаданных документа

* **Описание:** сохранить имя файла, тип, размер, длину текста, число страниц/абзацев при наличии.
* **Вход:** file + parsed text.
* **Выход:** document metadata.
* **DoD:** метаданные сохраняются вместе с документом и доступны на уровне API.

### PR-007 [P2] Чанкер текста для RAG

* **Описание:** разбивать текст на чанки с overlap.
* **Вход:** normalized text, chunk size, overlap.
* **Выход:** список чанков.
* **DoD:** чанки детерминированы; пустые и слишком короткие чанки не создаются.

## 5. Модуль промптов

### PM-001 [MVP] Реестр мастер-промптов

* **Описание:** вынести промпты в отдельный модуль с ключами и версионированием.
* **Вход:** ключ сценария генерации.
* **Выход:** текст шаблона и набор параметров.
* **DoD:** в коде нет хардкода промптов внутри endpoint/service файлов.

### PM-002 [MVP] Мастер-промпт прямой генерации квиза

* **Описание:** создать основной промпт для direct-mode генерации по документу.
* **Вход:** текст документа, число вопросов, сложность, тип теста.
* **Выход:** prompt payload.
* **DoD:** промпт требует русский язык, работу строго по документу и возврат по JSON Schema.

### PM-003 [MVP] Мастер-промпт восстановления невалидного JSON

* **Описание:** создать отдельный промпт для repair-pass.
* **Вход:** исходный невалидный JSON, сообщение об ошибке схемы.
* **Выход:** исправленный JSON.
* **DoD:** repair-pass не генерирует квиз заново, а чинит структуру.

### PM-004 [P2] Мастер-промпт для RAG-режима

* **Описание:** создать промпт, который работает с retrieved context, а не со всем документом.
* **Вход:** retrieved chunks, параметры генерации.
* **Выход:** prompt payload.
* **DoD:** промпт явно запрещает использовать знания вне переданного контекста.

### PM-005 [P2] Мастер-промпт регенерации одного вопроса

* **Описание:** создать промпт для пересоздания одного вопроса по фрагменту документа.
* **Вход:** предыдущий вопрос, retrieved context или document excerpt.
* **Выход:** один вопрос по схеме.
* **DoD:** результат включает только один вопрос и проходит ту же валидацию.

## 6. Модуль генерации

### GN-001 [MVP] Сборщик запроса на генерацию

* **Описание:** собрать полный payload для LM Studio из доменной команды генерации.
* **Вход:** `GenerationRequest`, текст документа, prompt template.
* **Выход:** provider request.
* **DoD:** inference params, схема ответа, системный и пользовательский промпт формируются в одном месте.

### GN-002 [MVP] Orchestrator прямой генерации

* **Описание:** реализовать pipeline `parse -> normalize -> prompt -> LM Studio -> normalize -> validate -> persist`.
* **Вход:** `document_id`, `GenerationRequest`.
* **Выход:** `Quiz`.
* **DoD:** pipeline не содержит логики endpoint-слоя; все шаги покрыты unit-тестами через моки.

### GN-003 [MVP] Контроль качества выходной структуры

* **Описание:** добавить post-check после генерации.
* **Вход:** `Quiz`.
* **Выход:** валидный квиз или ошибка качества.
* **DoD:** отлавливаются пустые вопросы, пустые ответы, несовпадение числа вопросов, дубликаты.

### GN-004 [MVP] Repair-loop для невалидного ответа модели

* **Описание:** если ответ модели не проходит схему, запускать ограниченный repair-pass.
* **Вход:** сырой ответ модели и ошибка валидации.
* **Выход:** исправленный квиз или финальная ошибка.
* **DoD:** число повторов ограничено; бесконечных циклов нет.

### GN-005 [MVP] Сохранение результата генерации

* **Описание:** сохранять квиз, исходный запрос, используемую модель и prompt version.
* **Вход:** `Quiz`, metadata.
* **Выход:** persisted quiz.
* **DoD:** по `quiz_id` можно восстановить, как был сгенерирован результат.

### GN-006 [P2] Генерация с потоковым статусом

* **Описание:** добавить статус генерации `queued/running/done/failed`.
* **Вход:** запрос генерации.
* **Выход:** status model.
* **DoD:** UI может опрашивать статус; зависшая генерация завершается timeout-ошибкой.

### GN-007 [P2] Адаптивный выбор режима direct vs RAG

* **Описание:** реализовать rule-based selector режима генерации.
* **Вход:** длина документа, число страниц, выбранный режим пользователем.
* **Выход:** `direct` или `rag`.
* **DoD:** ручной режим имеет приоритет; автоселектор детерминирован и покрыт тестами.

## 7. RAG-подсистема

### RAG-001 [P2] Генерация embeddings для чанков

* **Описание:** получить embedding-векторы чанков через LM Studio.
* **Вход:** список чанков.
* **Выход:** список `{chunk_id, vector}`.
* **DoD:** для каждого чанка есть вектор; ошибки отдельных чанков не ломают всю индексацию без контролируемой причины.

### RAG-002 [P2] In-memory индекс векторов

* **Описание:** реализовать минимальный локальный индекс для similarity search.
* **Вход:** пары `chunk + vector`.
* **Выход:** индекс документа.
* **DoD:** индекс можно построить, сохранить в памяти и использовать повторно внутри одной сессии.

### RAG-003 [P2] Retriever top-k

* **Описание:** реализовать поиск top-k чанков по embedding-вектору запроса.
* **Вход:** query text, document index, `k`.
* **Выход:** список релевантных чанков.
* **DoD:** retrieval детерминирован при одинаковых входах; пустой индекс корректно обрабатывается.

### RAG-004 [P2] Сборщик контекста для RAG-генерации

* **Описание:** собрать финальный контекст из top-k чанков с лимитом длины.
* **Вход:** список retrieved chunks, token/text budget.
* **Выход:** final context block.
* **DoD:** контекст не превышает лимит; порядок чанков стабилен.

### RAG-005 [P2] Orchestrator RAG-генерации

* **Описание:** реализовать pipeline `parse -> normalize -> chunk -> embed -> retrieve -> generate -> validate`.
* **Вход:** `document_id`, `GenerationRequest`.
* **Выход:** `Quiz`.
* **DoD:** direct и rag реализованы как разные стратегии под одним интерфейсом.

### RAG-006 [Future] Кэширование embeddings и индекса

* **Описание:** избежать повторной индексации для неизменного документа.
* **Вход:** `document_hash`.
* **Выход:** cached index.
* **DoD:** при повторной генерации по тому же документу embeddings не считаются повторно.
